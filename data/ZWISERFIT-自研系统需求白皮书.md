## 战略定位

本白皮书是 ZWISERFIT 从 MVP（双框架 AI Agent 军团）跃迁至自研系统的技术纲领。

> **"在 MVP 阶段是纲，在自研阶段是蓝图。"** — Shuyu 总指挥

**核心目标：**
1. 将已验证的 AI 协同流程转化为可执行的功能需求文档
2. 设计自研系统的核心模块划分，实现业务逻辑与 AI 调度框架的解耦
3. 建立战略预警机制，在关键阈值触发时自动建议启动自研

## 第二部分：自研系统目标架构

### 2.1 核心设计原则

```
┌─────────────────────────────────────────────────┐
│            ZWISERFIT 自研系统 设计原则            │
├─────────────────────────────────────────────────┤
│ 1. 业务逻辑 ≠ AI 调度框架（彻底解耦）               │

│ 5. 数据主权完全归属 ZWISERFIT，零外部依赖            │
│ 6. 开发体验优先（新开发者 1 天内读懂核心脉络）        │
└─────────────────────────────────────────────────┘
```

### 2.3 核心模块拆解（MVP → 自研）

#### 模块 A：会员管理服务（Member Service）

| 维度 | MVP（当前） | 自研目标 |
|------|-----------|---------|
| **数据存储** | 内存/文件系统 | PostgreSQL |
| **身份认证** | 无（单用户） | OAuth2.0 + JWT |
| **门店绑定** | 无 | 门店-会员 N:1 关系 |
| **档案管理** | 创始人手动维护 | RESTful API + 后台管理 |
| **性能目标** | N/A | 支持 10,000+ 会员并发查询 |
| **优先级** | — | 🔴 P0（核心） |

**功能需求（FR-MEMBER）：**
- FR-MEMBER-001：会员注册（手机号/微信/邮箱）
- FR-MEMBER-002：会员档案 CRUD（健康画像、行为标签）
- FR-MEMBER-003：门店绑定与解绑
- FR-MEMBER-004：会员等级体系（基于 ZWF-20 积分）
- FR-MEMBER-005：数据脱敏中间件（ZWF-20 标准）

#### 模块 B：数据采集服务（Collector Service）

| 维度 | MVP（当前） | 自研目标 |
|------|-----------|---------|
| **数据来源** | Agent 手动报告 | 标准化 API / SDK |
| **数据格式** | Markdown / JSONL | Protobuf / JSON Schema |
| **脱敏处理** | Prompt 指令 | 独立脱敏中间件 |
| **实时性** | 准实时（分钟级） | 实时流处理 |
| **优先级** | — | 🔴 P0（核心） |

**功能需求（FR-COLLECTOR）：**
- FR-COLLECTOR-001：健康行为数据标准化采集 API
- FR-COLLECTOR-002：数据质量校验（ZWF-20 Schema）
- FR-COLLECTOR-003：自动脱敏管道（姓名/手机/身份证 → 匿名ID）
- FR-COLLECTOR-004：数据血缘追踪（来源 → 处理 → 存储全链路）
- FR-COLLECTOR-005：异常数据告警

#### 模块 C：积分权益服务（Points Service）

| 维度 | MVP（当前） | 自研目标 |
|------|-----------|---------|
| **积分规则** | 概念阶段 | 可配置规则引擎 |
| **权益兑换** | 无 | RESTful API |
| **审计** | Stella 手工审计 | 自动审计日志 |
| **优先级** | — | 🟡 P1（重要） |

**功能需求（FR-POINTS）：**
- FR-POINTS-001：积分规则配置（行为→积分映射）
- FR-POINTS-002：积分发放/扣减（事务性）
- FR-POINTS-003：权益目录与兑换
- FR-POINTS-004：积分账本（不可篡改审计链）
- FR-POINTS-005：风控规则（异常积分行为检测）

#### 模块 D：合约履约服务（Contract Service）

| 维度 | MVP（当前） | 自研目标 |
|------|-----------|---------|
| **RWA 封装** | 概念/文件 | API 驱动的标准化封装 |
| **履约验证** | 手工 | 自动验证 + 链上存证 |
| **优先级** | — | 🟡 P1（重要） |

**功能需求（FR-CONTRACT）：**
- FR-CONTRACT-001：RWA 资产包标准封装（ZWF-20 Schema）
- FR-CONTRACT-002：智能合约模板管理
- FR-CONTRACT-003：履约条件自动验证
- FR-CONTRACT-004：链上存证接口（多链适配）
- FR-CONTRACT-005：资产包生命周期管理

#### 模块 E：AI 调度引擎（AI Orchestrator）

| 维度 | MVP（当前） | 自研目标 |
|------|-----------|---------|
| **Agent 管理** | OpenClaw 框架 | 自研编排引擎 |
| **Prompt 管理** | 散落各 Agent | 统一 Prompt 仓库 + 版本管理 |
| **模型路由** | 单一模型 | 多模型智能路由（成本/质量/延迟） |

## 第五部分：战略预警机制

### 5.1 预警阈值定义

| 预警指标 | 黄灯阈值 🟡 | 红灯阈值 🔴 | 当前值 | 检查频率 |
|---------|:---:|:---:|------|:---:|
| 活跃门店数 | ≥50 | ≥200 | 待采集 | 每周 |
| 注册用户数 | ≥1,000 | ≥10,000 | 待采集 | 每周 |
| 月度 API 总成本 | ≥¥5,000 | ≥¥15,000 | 待采集 | 每周 |
| OpenClaw 框架阻塞 | 1个核心模块 | 3个核心模块 | 0 | 实时 |
| Cron Job 超时率 | ≥30% | ≥50% | 多次 | 每日 |
| 单 VM 资源使用率 | CPU≥70% | CPU≥90% | ~50% | 每日 |

**成本对照：**
- 月度 API + 服务器成本 > ¥50,000 → 超过最小开发团队月薪（¥120K）的 40% → 触发黄灯
- 月度 API + 服务器成本 > ¥100,000 → 触发红灯，正式建议启动自研

### 5.2 预警触发后的标准动作

1. **自动生成《自研系统可行性评估简报》**
2. **推送到 Shuyu 总指挥的企业微信**
3. **简报包含：**
   - 触发指标及当前值 vs 阈值
   - 当前月度成本 vs 自研团队月度成本对比

   - 预计开发周期（基于模块优先级）
   - 风险评估（技术风险 + 业务风险 + 人力风险）
   - 推荐行动：立即启动 / 准备启动 / 继续观察

### 5.3 框架阻塞定义

以下情况视为"在第三方框架上遇到无法逾越的墙壁"：
1. Agent 上下文窗口限制导致核心业务流程无法完整执行
2. 框架不支持多租户隔离（数据安全红线）
3. 框架性能瓶颈无法通过垂直扩展解决
4. 框架缺失的关键能力无法通过插件机制补充
5. 框架许可证变更或供应商锁定风险

## 第六部分：技术选型建议

### 6.1 推荐技术栈

| 层级 | 推荐方案 | 备选方案 | 选择理由 |
|------|---------|---------|---------|
| **后端语言** | Go | Java (Spring) | 高性能、低资源消耗、适合中小团队 |
| **微服务框架** | Go-Zero / Kratos | Spring Cloud | 国产成熟框架，社区活跃 |
| **API 网关** | Kong | APISIX | 插件生态丰富，K8s 原生 |
| **消息队列** | NATS | RabbitMQ | 轻量级，Cloud Native |
| **业务数据库** | PostgreSQL | MySQL 8.0 | JSON 支持好，RWA 场景适合 |
| **缓存** | Redis | — | 行业标准 |
| **对象存储** | MinIO | 腾讯云 COS | 自部署，数据主权 |
| **容器编排** | Docker Compose → K8s | — | 渐进式迁移 |
| **可观测性** | Prometheus + Grafana | — | 开源标准 |
| **AI 模型** | DeepSeek API → 自部署 | Qwen / Claude | 逐步降低外部依赖 |
| **前端** | React + Ant Design | Vue3 + Element Plus | 管理后台生态成熟 |

### 6.2 关键决策：语言选择 Go vs Java

| 维度 | Go | Java (Spring) |
|------|:---:|:---:|
| 学习曲线 | 🟢 低 | 🟡 中 |
| 内存占用 | 🟢 极低（10-50MB/服务） | 🔴 高（200-500MB/服务） |
| 开发速度 | 🟢 快 | 🟡 中 |
| 人才市场 | 🟡 增长中 | 🟢 存量最大 |
| 微服务生态 | 🟢 Go-Zero/Kratos 成熟 | 🟢 Spring Cloud 最成熟 |
| AI/ML 集成 | 🟡 通过 API | 🟢 直接调用 Python |
| **推荐** | ⭐ **首选** | 备选 |

> **建议：** Go 为主力语言，AI 密集模块通过 sidecar 模式调用 Python 服务。
