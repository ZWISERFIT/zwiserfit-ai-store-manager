## 二、根因分析

### 2.1 主因：DeepSeek API 严重延迟

```
models.list API 响应时间: 35,000 - 41,000ms（正常 < 1,000ms）
```

WeCom 消息已通过 WebSocket 推送到 Gateway，但 Agent 调用 DeepSeek API 时被阻塞。高延迟→Agent 超时→用户收不到回复。

**确认方式**：
- Gateway 日志 `models.list` 耗时记录
- 网关重启后 WeCom WebSocket 5 个账号全部认证成功（排除连接问题）
- `curl api.deepseek.com` 修复后 TTFB 119ms（API 已恢复）

### 2.2 次因：飞书 undici 崩溃风暴

```
Feishu [default] channel exited: Cannot find package 'undici'
Feishu 崩溃次数: 74 次（exit）
Feishu 重启尝试: 74 次（auto-restart 1/10 → 8/10）
```

飞书通道每次崩溃后尝试重连（指数退避 83s → 300s），持续消耗 CPU 和内存资源，加剧事件循环压力。

### 2.3 连带影响：事件循环阻塞

| 指标 | 修复前（峰值） | 修复后 | 改善 |
|------|-------------|--------|------|
| `eventLoopDelayMaxMs` | **1,124,207ms**（18.7分钟） | 1,372-6,446ms | **99.9% ↓** |
| `eventLoopUtilization` | 0.83-0.85（83-85%） | 0.04-0.07（4-7%） | **16x ↓** |
| `cpuCoreRatio` | 0.83-0.85 | 0.05-0.07 | **14x ↓** |
| `liveness warnings` | 40 次/小时 | ~0 次（无新告警） | ✅ 消除 |

## 三、修复方案

### Step 1：禁用飞书（止损）

```json
// channels.feishu.enabled: true → false
```

**原理**：飞书 undici 缺失是历史遗留依赖问题，短期不可修复。先禁用释放资源，聚焦企微恢复。

**耗时**：1 分钟（编辑 + SIGUSR1 热加载）

### Step 2：DeepSeek API 延迟验证

```bash
curl -s -o /dev/null -w "Connect: %{time_connect}s | TTFB: %{time_starttransfer}s | Total: %{time_total}s" \
  --connect-timeout 10 https://api.deepseek.com/v1/models
```

**结果**：Connect 7ms / TTFB 119ms → API 已恢复，非永久性故障。

### Step 3：写入模型降级配置

```json
{
  "agents": {
    "defaults": {
      "model": {
        "primary": "deepseek/deepseek-v4-pro",
        "fallbacks": ["deepseek/deepseek-chat"]
      }
    }
  }
}
```

**原理**：当 `v4-pro` 超时或不可用时，自动切换到 `chat` 模型，避免再次出现全部 Agent 阻塞。

**⚠️ 格式教训**（Baron 纠正）：
- ❌ 错误：`"model": "deepseek/deepseek-v4-pro", "fallbacks": [...]`
- ✅ 正确：`"model": { "primary": "...", "fallbacks": [...] }`
- 降级配置必须将 model 字段从字符串改为对象，`primary` 和 `fallbacks` 是 model 对象的子键

## 四、关键指标记录

### 故障特征（用于未来自动检测）

| 指标 | 阈值 | 含义 |
|------|------|------|
| `models.list` 响应时间 | > 5,000ms | DeepSeek API 可能延迟 |
| `eventLoopDelayMaxMs` | > 30,000ms | 严重事件循环阻塞 |
| `eventLoopUtilization` | > 0.8 持续 5 分钟 | CPU 过载 |
| `channel exited` 日志 | > 5 次/10 分钟 | 通道崩溃风暴 |
| `liveness warning` | > 3 次/15 分钟 | 需要立即排查 |

### 通道隔离原则

```
企业微信（WebSocket） ≠ 飞书（WebSocket） ≠ Hermes（独立系统）

一个通道的故障不应影响其他通道，但：
  - 共享进程的通道（飞书/企微）会通过事件循环互相影响
  - 独立系统的通道（Hermes/Momo）天然隔离
```
